'a:1:{i:0;s:36:"New link: Goole http://goo.gl/zIfjp";}' --- Expression are pretty useful, many of us use them really often. Introduced for LINQ there are many other use cases for example compiler check able Propertychange events. If you ask for how to implement INotifyPropertyChange on Stackoverflow you will get answers like that. http://mfelicio.com/2010/01/10/safe-usage-of-inotifypropertychanged-in-mvvm-scenarios/ But not many people think about the cost they add at runtime. I must admit I didn't, too. But in the last weeks I was involved into a new project at a customer, which took that to my attention.  The project is huge .Net C# project which was started back in the times where .Net 2 was released and the purposed Microsoft Design pattern for such Applications was CAB <todo add CAB link>. The project is about 7M PLOC and suffers some problems in terms of  maintainability and performance.  In one of the project meetings a discussion about using expressions for creating Bindings and publishing NotifyPropertyChanged events came up. Since there are allready some performance issues the killer argument against using it was that it adds a significant performance overhead.  But noone was able to come up with some numbers and their real world impact on the application performance. Some argued that expressions are just some special string that compiler checks. Well lets proove whats really going on: I started with a simple test how long it takes to do some expression evaluation like you would do when you want to get the member name from an expression. With some simple code like this: 
        private static string Test(Expression> expression )
        {
            return GetMemberName(expression);
        }

        private static string GetMemberName(Expression expression)
        {
            var sw = Stopwatch.StartNew();

            var lambda = expression as LambdaExpression;

            if (lambda == null )
                throw new  ArgumentException();

            var member = lambda.Body as MemberExpression;

            if(member == null)
                throw new ArgumentException();

            sw.Stop();

            ExpressionEvalTimes.Add(sw.ElapsedTicks);

            return member.Member.Name;

        }
And the result was some kind of suprising.  About ~45 Ticks per call, thats a significant overhead, but since it is just a pretty simple implementation I wanted to looks a bit closer to it if it is possible to speed it up a bit. Lets check the pure evaluation time:

Huh? Only ~5 Ticks for evaluation? Looks like we are loosing the most of the time at some other place, time to have a look at the resulting IL Code to see whats going on under the hood. The IL Code looks like this:

As you can see some reflection is involved to create the expression. And as we know Reflection is slow, the metadate lookup take a lot of time and it is done everytime we call the method that takes a expression as parameter. And it is done twice, once for looking up the type of the expression and the second time to get the member that we express with the expression. Ok now we know what the theoretical overhead is, but what does that mean in the real world? Is it really significant? Lets do some testing that is closer to the real world.
